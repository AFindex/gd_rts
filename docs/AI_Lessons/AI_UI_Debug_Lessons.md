# AI UI Debug Lessons (2026-03-03)

## 0. 2026-03-05 增量教训（手动布局版）

### 0.1 `none -> multi` 过渡抖动根因（已修复）

1. **不要用 `visible=false` 做“首帧过渡隐藏”**  
在手动布局链里，`visible=false` 常会让控件先进入塌缩尺寸（本次 `MultiMatrixRoot` 先到高度 4），下一步再恢复正常高度，玩家会看到“矩阵跳位”。

2. **正确做法是“保持可布局，视觉隐藏”**  
过渡帧用 `self_modulate.a=0`（必要时 `mouse_filter=IGNORE`）隐藏视觉，但保留节点参与完整布局；reveal 时只恢复透明度，不引入额外尺寸跃迁。

3. **状态时序要先更新再布局**  
`update_hud` 内应先刷新当前模式标记，再进入布局流水线；否则该帧会用旧模式执行布局，容易触发错误分支（例如把 multi 当 none 处理）。

1. **`PanelContainer` 的可用内容区不能用 `panel.size` 直接算**  
手动布局时应扣掉 `StyleBox` 四边 margin，再给子节点布局；否则很容易出现“看起来尺寸对，子内容却挤压/错位”。

2. **首帧尺寸不稳定时，手动布局会得到 0 高/异常高**  
HUD 初始化后应做短暂 warmup（前几帧强制 reflow），避免 `QueueContent/CommandGrid` 在未稳定尺寸上完成错误布局。

3. **`custom_minimum_size` 会悄悄对抗手动布局**  
即使代码里设置了 `Control.size`，最小尺寸仍可能把格子撑坏或重叠；命令格/矩阵格需要显式清零或统一约束。

4. **“固定最小格子”不适合高密度 RTS 面板**  
命令卡、多选矩阵应优先按可用区域 fill 计算，再做居中；固定小格更容易造成“堆在角落”和视觉过小。

5. **`call_deferred` 刷新链必须防重入**  
布局函数内部再次排队刷新会造成刷新风暴，严重时触发引擎崩溃（本次出现 `signal 11`）；要有 `_is_applying_manual_layout` 保护。

6. **排障要先看运行时实际几何，不猜**  
用临时探针打印关键节点 `position/size`，比肉眼判断快很多，尤其适合定位“为什么某个容器始终 0 高”。

### 0.2 容器级扩列 vs 内容级扩列（2026-03-05）

1. **“扩 Matrix 一列”必须先改容器分配，不要只改 `QueueContent` 内宽**  
如果只在内容层做 `+width`，视觉上会出现“内容假扩宽”，但 `Portrait/Command` 容器宽度和位置不变，最终布局关系仍然不对。

2. **同一需求要在同一层闭环**  
本次正确做法是：在 `BottomRow` 四列分配阶段做重分配，再让 `QueueContent` 只填充自身容器。  
也就是“列宽变化在列级解决，内容层只负责填充”。

3. **固定约束要显式写死**  
右侧两列约束应保持为：  
`PortraitColumn` 宽度不变，仅右移；  
`CommandColumn` 贴右边缘并收窄；  
`QueueColumn` 向右扩展。

4. **扩列增量应与列数联动，不要写一次性常量**  
推荐公式：`queue_expand = max(0, manual_matrix_columns - 8) * medium_button_width`。  
这样从 8->9->10 列时，容器重分配会连续、可预测。

5. **避免“双重扩宽”**  
容器级已经扩过后，`QueueContent` 不能再额外 `+width`，否则等于重复加宽，会导致后续比例失真和对齐偏移。

## 1. 本次对话目标与结果

本轮对话围绕 `RTS HUD` 布局问题展开，核心诉求包括：

- `BottomHUD` 高度与位置异常（占屏过大、下方溢出）。
- `QueuePanel` / `PortraitPanel` 希望比两侧面板更矮。
- 非主面板的占位/容器节点希望透明，不参与视觉表达。
- `CommandGrid` 运行时按钮布局异常（宽度过窄、高度被拉爆）。

最终确认有效的修复方向：

- 以 `.tscn` 作为布局真源，减少/去除运行时布局重写。
- `CommandGrid` 问题通过两点稳定修复：
  - 运行时按钮 `custom_minimum_size` 增加宽度下限。
  - `CommandGrid` 去掉垂直 `EXPAND`，避免高度被过度拉伸。

## 2. 关键过程复盘

### 阶段 A：快速收敛 `BottomHUD` 溢出

- 先通过 `BottomHUD` 高度与内部最小尺寸（小地图、命令按钮）做减压，短期改善了半屏占用与溢出。
- 后续按参考图继续做比例拟合时，尝试过较多“运行时自适应”逻辑。

### 阶段 B：复杂自适应引入副作用

- 使用运行时逻辑动态改 `offset/custom_minimum_size/spacing` 时，出现了“场景值与运行时值相互覆盖”的感知问题。
- 多轮改动后，用户反馈“布局变乱/改了像没改”，说明可预测性下降。

### 阶段 C：切回容器化与可控最小改动

- 针对“中间两块更矮”尝试了 `QueueColumn/PortraitColumn + TopSpacer` 模式。
- 这个模型可表达需求，但若同时叠加运行时重算，依然容易和静态布局冲突。

### 阶段 D：明确单一真源

- 用户明确希望“直接改 HUD 内容”，因此转为：
  - 场景内固定 `BottomHUD` 尺寸与位置。
  - 清理运行时布局相关代码，只保留数据更新与样式逻辑。

### 阶段 E：定位 `CommandGrid` 异常

- 观测到按钮 `x` 极小、`y` 极大。
- 在“先不动列数与清空机制”的前提下，只修两项：
  - 按钮最小宽度不为 0。
  - `CommandGrid` 不再垂直扩展填满父容器。
- 结果有效，问题被用户确认修复。

## 3. 值得记忆的教训

1. **UI 必须有单一真源**  
布局参数如果同时由 `.tscn` 和运行时代码控制，排查成本会指数上升，用户会感知为“改了没用”。

2. **先最小改动，再做体系化重构**  
用户给出明确修复路径时，应优先执行最小闭环，不要先引入大规模自适应体系。

3. **Godot 容器行为要先确认再下手**  
`HBoxContainer/GridContainer` 的尺寸分配规则经常是问题根因。误判容器规则会导致“表面参数对、结果仍错”。

4. **按钮尺寸不要给 0 宽下限**  
在拥挤容器中，`custom_minimum_size.x = 0` 很容易被压扁成几像素，直接造成“看似随机”的布局灾难。

5. **`EXPAND` 是高风险开关**  
在高度受限区域，`size_flags_vertical = EXPAND_FILL` 会把单元高度拉爆，尤其是 Grid 的行列不匹配时。

6. **先验证“是否真的被运行时覆盖”**  
遇到“看起来没生效”，第一步应检查是否有 `_ready/_process/size_changed` 在持续重写属性。

7. **透明辅助节点最好静态+动态双保险**  
对于占位/容器节点，必要时可在 `.tscn` 设透明，再在运行时兜底一次，防止主题或脚本覆盖。

## 4. 下次同类问题的推荐流程

1. 先确认“当前值来自哪”（场景还是代码）。
2. 把布局入口函数与监听器列全（`_ready`、`size_changed`、定时器、动画）。
3. 锁定一个真源（优先场景静态或单一布局函数二选一）。
4. 先修最小可复现点（单节点/单容器），验证后再扩展到全局。
5. 只在用户确认后再做自适应体系化改造。

## 5. 本轮可复用结论（针对本项目）

- `RTS HUD` 当前更适合“场景静态布局 + 脚本只做数据渲染”。
- `CommandGrid` 若再异常，优先检查：
  - 按钮最小宽高；
  - `CommandGrid` 的 `size_flags_vertical`；
  - 父容器可用宽高与分栏比。
